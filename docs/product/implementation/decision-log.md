This made `closeDatabase` `async`, and every call site across `@iroha/core`/`@iroha/storage`/their tests (~20 files) was updated to `await` it. The existing `renameWithRetry`/rm-retry budgets from (12) are kept as defense-in-depth, not removed, since the DELETE-mode switch is not confirmed by any of the cited sources to fully eliminate the lock in every case (the bun issue's reporter did not test it, and cross-connection contention can still prevent the mode switch itself from taking effect) — this still needs a Windows CI run to confirm the actual effect. The product owner chose a **dedicated table via a new forward-only migration** over the two alternatives (a `local_settings` KV entry — no migration but mixes a security token into a general settings store; or deferring token issuance entirely to WP-07, which would leave SessionStart's contract output incomplete). `migrations/002_session_tokens.sql` (mirrored to `docs/product/migrations/`) adds `session_tokens` (`token_hmac` PRIMARY KEY with a `GLOB 'hmac-sha256:*'` CHECK, `repository_id`/`session_id`/`run_id` FKs with `ON DELETE CASCADE`, `platform` CHECK, `issued_at`/`last_used_at`/`expires_at`, plus indexes on `run_id` and `expires_at`), sets `PRAGMA user_version = 2`, and is self-contained (`BEGIN IMMEDIATE`/`COMMIT`) exactly like 001. Only the salt-keyed HMAC of the token is stored, never the plaintext, so a leaked DB yields no usable token. The table is disposable local operational state: `rebuildDatabase` imports canonical only, so `session_tokens` is empty after a rebuild — which is correct, since tokens are per-Run ephemera. `@iroha/storage` gains `insertSessionToken`/`getSessionToken` (the write + read-back surface WP-06 needs); token verification, `last_used_at` bump, and run-completion/expiry invalidation are WP-07's (the MCP server is the only token consumer) and are intentionally not built now (YAGNI). `database-schema.md` §5, `design.md` §7's table group, and `migrator.test.ts` (its "applies the real migration" assertions, and the synthetic-migration crash-recovery/downgrade tests that had used version 2 as a stand-in "unknown future version" — now shifted to version 3 to avoid colliding with the real 002) were all updated in the same change, per the change protocol. | Accepted |

## Change protocol

Accepted decisions require:

1. a new ADR in `../design.md`;
2. affected machine schema/migration change;
3. backward-compatibility and migration statement;
4. updated acceptance tests;
5. explicit product-owner approval when canonical data, privacy, public API, or distribution changes.

Implementation convenience alone is not sufficient reason to replace an accepted decision.
