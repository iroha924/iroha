| ID-022 | WP-02 (`@iroha/git`) implementation choices for details left open by prose: (1) the HMAC repository salt (32 random bytes, base64url-encoded) is stored as a `repositorySalt` field inside `<git rev-parse --git-path iroha>/local-config.json`, the local-only file design.md §6 already reserves for this purpose; reads/writes preserve unrelated fields so later work packages can add their own keys to the same file. Writes go through a temp-file-then-`rename` for atomicity; a concurrent-first-write race (two processes both seeing no salt and generating different values, last writer wins) is accepted as a rare, self-correcting condition rather than adding cross-process file locking, since no WP in `implementation-plan.md` scopes a locking primitive yet. None of these affect canonical data, security boundaries, or the public MCP/CLI surface; recorded here per the decision rule rather than blocking on human input. | Accepted |
| ID-023 | WP-02 (`@iroha/git`) `remote.ts`/`credential-redaction.ts` reached diminishing returns during PR review: after resolving several dozen confirmed-by-reproduction findings (credential leaks via multiple text-boundary shapes, local-path suppression, `--worktree`/`--includes`/`insteadOf`/multi-value config resolution), each fix increasingly targeted narrower Git CLI edge cases rather than remaining security-critical gaps, and one fix round (the `--worktree` fallback) introduced its own regression (exit 128 for `extensions.worktreeConfig` disabled + multiple worktrees) that a later round had to catch — confirming the marginal fixes were adding as much risk as they removed. Confirmed with the product owner (chose to stop chasing further PR-review-discovered edge cases rather than open-endedly harden Git config parsing). | Accepted |
| ID-024 | WP-03 (`@iroha/storage`) implementation choices for details left open, or ambiguous, in prose: (1) `database-schema.md` §4 says migration files are named `<four-digit>_<name>.sql`, but the only actual migration file, `migrations/001_initial.sql`, uses a 3-digit prefix, and every other cross-reference in the doc bundle (`design.md`, `README.md`, this file's own ID-020/ID-021 entries, `validation-report.md`) also cites `001_initial.sql`. Rather than picking a side of that prose/filename disagreement, `migrator.ts`'s filename regex (`/^(\d+)_(.+)\.sql$/`) accepts any digit-count numeric prefix, satisfying both readings without contradicting either. (2) §3's "`PRAGMA busy_timeout = 2500`" and "retry SQLITE_BUSY with bounded jitter for at most 2 seconds" read as two independent rules, but confirmed by reproduction: a single blocked `@libsql/client` call under that PRAGMA is a synchronous native call that stalls this process's whole event loop for up to ~2.5s before ever returning control to JS — so if the 2-second app-level retry budget were counted from before that first attempt, it would already be exhausted by the time the first failure is observed. `transaction.ts`'s `withTransaction` therefore starts its 2-second retry-budget clock only after the first SQLITE_BUSY failure, treating the PRAGMA's own wait and the app-level retry budget as sequential rather than overlapping. (3) `database-schema.md` §11 says "Recursive CTE queries must track visited IDs in the path and enforce both depth and edge limits" for `getNeighbors`/`getPath`/`getSubgraph`; `graph-search.ts` implements the latter two as breadth-first search in application code (one `getNeighbors` SQL call per BFS frontier node) rather than a single recursive SQL CTE, since SQLite recursive CTEs do not guarantee BFS expansion order without a hand-built "visited path as string" workaround, and a local libSQL round-trip is cheap enough that a plain, easily-verified BFS was judged simpler and more trustworthy (same required properties — bounded depth, edge limit, cycle avoidance via a visited-entity set — are preserved). (4) Of `database-schema.md` §13's application-level integrity checks, `integrity.ts`'s `checkIntegrity()` implements only the ones expressible purely against this database: approved-knowledge-without-canonical-document, FTS-index/`search_documents` drift (via each FTS5 table's `_docsize` shadow table — confirmed by reproduction that a plain `count(*)`/unfiltered `SELECT` against an external-content FTS5 table always mirrors its content table's row range regardless of actual index sync state, so it cannot detect this drift), stale embedding `content_hash`, and relations pointing to a rejected entity. Checks needing filesystem access to canonical files (path existence, filename/ID agreement) are left to the CLI/sync layer that has that access, and "no canonical ID represented by multiple paths" is skipped as already structurally guaranteed by `canonical_documents.entity_id` being its primary key. None of these affect canonical data, security boundaries, or the public MCP/CLI surface; recorded here per the decision rule rather than blocking on human input. (5) `migrator.ts`'s `getAppliedMigrations` (read) and `db.executeMultiple(file.sql)` (write) are not atomic across OS processes; two processes both calling `runMigrations` against the same not-yet-migrated file at once race on the migration's own `CREATE TABLE`, confirmed by reproduction to surface as a generic `INTERNAL_ERROR` (`SQLITE_ERROR: table ... already exists`, a code `errors.ts` does not classify as `DB_BUSY`) rather than a distinguishable retry signal — though re-running `runMigrations` afterward cleanly reports `ok([])`. This is accepted as a rare, self-correcting race (only reachable during the very first `iroha init` against a given repository, at the exact instant two processes both run it) rather than adding cross-process locking, matching ID-022's identical reasoning for the repository-salt write race: no WP in `implementation-plan.md` scopes a locking primitive yet. (6) `database-schema.md` §13 lists eight application-level integrity checks; `checkIntegrity()` implements four (point 4 above) and explicitly excludes two more (path-existence/filename-ID agreement, deferred to the CLI/sync layer; no-multiple-paths, structurally guaranteed). The remaining one, "every active Guardrail has a valid guard spec," needs no filesystem access and so does not fit either exclusion reason — it is omitted because no guard-spec JSON Schema exists anywhere in the repository yet (`schemas/` has no `guard-spec` definition; `hooks-contract.md`/`mcp-contract.md` reference "a machine-evaluable guard spec" without defining its concrete shape), so "valid" beyond well-formed JSON is not yet checkable; `knowledge_items`'s own `CHECK` constraints already guarantee `guard_spec_json` is present and well-formed JSON when `enforcement = 'guardrail'`. Revisit once a guard-spec schema exists (likely alongside Guardrail-authoring work, not yet scoped to a WP). | Accepted |
| ID-025 | WP-04 (`@iroha/config`, `@iroha/canonical`) implementation choices for details `compatibility.md`/`canonical-schema.md` leave open: (1) library choices, none named anywhere in the doc bundle or pnpm catalog before this WP: `yaml` (eemeli/yaml) for all YAML parsing (frontmatter, `config.yaml`, `labels.yaml`) as the actively-maintained ESM/TypeScript-native standard; `mdast-util-from-markdown`/`mdast-util-to-string`/`unist-util-visit` for §7 body-template heading validation over hand-rolled line/regex scanning — confirmed by reproduction that a naive `#`-prefix regex misidentifies a heading-shaped line inside a fenced code block, which a real CommonMark AST parser correctly ignores; `secretlint`/`@secretlint/node`/`@secretlint/secretlint-rule-preset-recommend` for §11 step 2 / §12 step 3 secret scanning over hand-rolled regex patterns, chosen because it is a maintained, broad-coverage rule set with a programmatic `executeOnContent({content, filePath})` API (confirmed by reproduction) that scans an in-memory string before any file write, not just files on disk — and because `iroha` ships as an npm CLI that must run secret scanning on arbitrary end-user machines without assuming an external binary (e.g. `gitleaks`) is installed, unlike this repo's own CI-only `secrets-scan` job. (2) `@iroha/canonical`'s contribution to §12's "approval transaction" is scoped to steps 3-7 (Zod/body-template/secret validation, write to sibling temp file, flush, atomic rename, fsync parent directory) only — steps 1/2/8/9/10 (per-repository write lock, candidate reload/concurrency-token check, local DB entity/relation/search commit, approval audit record, unlock) need `@iroha/storage`/`@iroha/git`, which `compatibility.md` §4 does not allow `@iroha/canonical` to depend on (`domain`, `config` only). A not-yet-built higher layer (CLI/dashboard-api) composes `writeCanonicalDocument`'s result with those packages to complete the full transaction. (3) `computeCanonicalPath` derives the on-disk path (§3-4: type-specific subdirectory, `sessions/YYYY/MM/` from UTC `created_at`, `<id>.md` basename) directly from the validated document's own `id`/`type`/`created_at`, rather than accepting and validating a caller-supplied path — eliminating an entire class of caller/document path-mismatch bugs instead of adding a separate check for it, since `id` has already passed Zod's strict ULID pattern before it is used to build a path (no `..`-traversal risk per `.claude/rules/path-and-symlink-safety.md`). (4) `parseCanonicalDocument` strictly rejects CRLF line endings and a UTF-8 BOM (§5's format contract) but leniently strips *any* number of leading/trailing blank lines around the body (not exactly one), matching what `serializeCanonicalDocument` always produces while tolerating minor hand-edit whitespace variance that is not a substantive contract violation; the round-trip check in `serializeCanonicalDocument` (§11 steps 9-10) uses structural (`node:util`'s `isDeepStrictEqual`) rather than string/JSON equality, since Zod's re-parsed object is not guaranteed to preserve the exact key insertion order of the hand-ordered candidate even when semantically identical. None of these affect canonical data format, security boundaries, or the public MCP/CLI/dashboard-api surface (not yet built); recorded here per the decision rule rather than blocking on human input. | Accepted |

## Change protocol

Accepted decisions require:

1. a new ADR in `../design.md`;
2. affected machine schema/migration change;
3. backward-compatibility and migration statement;
4. updated acceptance tests;
5. explicit product-owner approval when canonical data, privacy, public API, or distribution changes.

Implementation convenience alone is not sufficient reason to replace an accepted decision.
